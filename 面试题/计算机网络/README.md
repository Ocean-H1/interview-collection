## 1.正向代理和反向代理的区别？应用场景？

| **特性**       | **正向代理**                     | **反向代理**                 |
| -------------- | -------------------------------- | ---------------------------- |
| **代理对象**   | 代理客户端                       | 代理服务器                   |
| **客户端感知** | 客户端需主动配置代理             | 客户端无感知                 |
| **隐藏对象**   | 隐藏客户端 IP                    | 隐藏服务器 IP                |
| **应用场景**   | 突破访问限制、匿名访问、缓存加速 | 负载均衡、安全防护、动静分离 |
| **配置位置**   | 客户端侧                         | 服务器侧                     |

简单示例:

```nginx
# 正向代理
# nginx.conf
server {
    listen 80;
    resolver 8.8.8.8;  # 必须指定DNS解析器

    location / {
        proxy_pass http://$http_host$request_uri;  # 透传原始请求
        proxy_set_header Host $http_host;
    }
}
```

```nginx
# 基础反向代理
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend_servers;  # 转发到后端服务器组
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

# 定义后端服务器组（负载均衡）
upstream backend_servers {
    server 192.168.1.101:8080 weight=3;  # 权重负载
    server 192.168.1.102:8080;
    server backup.example.com:8080 backup;  # 备用服务器
}

```

## 2.HTTP/1.0~3.0发展历史?(解决了哪些痛点，进行了什么优化)

| HTTP 版本 | 主要特点                                                                                                                                   | 主要痛点                                                                                     |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| 1.0       | **短链接**：每次请求都需要建立新连接                                                                                                       | 速度慢，开销大，效率低                                                                       |
| 1.1       | **默认长连接**：一条TCP连接，可以发送多个请求/响应<br />**增强缓存机制**<br />**Host请求头**：支持一台服务器部署多个站点<br />**断点续传** | **队头阻塞**：同同一连接里前一个请求卡住，后面所有请求都会被阻塞<br />**文本协议，头部冗余** |
| 2.0       | **二进制分帧**：可乱序、穿插传输，同时解析效率更高<br />**多路复用**<br />**HPACK头部压缩**<br />**服务器推送**                            | 受到TCP底层限制，**队头阻塞问题并未彻底解决**(丢包会导致整个连接阻塞)                        |
| 3.0       | **放弃TCP，使用QUIC协议(基于UDP)**：<br />彻底消除队头阻塞<br />支持0-RTT握手，建联更快<br />默认加密，安全性更高                          | /                                                                                            |

## 3.HTTP2中，二进制分帧的核心价值是？

> 通过二进制分帧将请求/响应拆分成带stream id 的帧，是多路复用的前提(支持乱序，交错传输)。同时，由于放弃HTTP/1.X的纯文本，改用二进制，所以解析更快，传输效率更高

## 4.HTTP2中，多路复用的核心价值？

> 只需要1条TCP连接，就能并行请求。减少建连开销，不再需要浏览器开6-8个连接来优化

## 5.HTTPS使用的是非对称加密还是对称加密？

> HTTPS采用混合加密(对称加密 + 非对称加密)方式：使用非对称加密传输对称加密所需的密钥。这样既能保证密钥在传输过程中的安全，又节省性能

## 5.简述HTTPS握手过程？

> HTTPS首先通过三次握手建立TCP连接之后，在进行TLS握手(标准情况)：
> 客户端发送支持的加密套件(加密算法、哈希算法等)和随机数
> 服务端返回证书，公钥以及选定的加密套件
> 客户端验证证书，生成预主密钥并用公钥加密后发送给服务端
> 两端用三个随机数算出对称会话密钥

## 5.有哪些常见的网络攻击？如何进行防范？

| 攻击名称                | 核心原理                                                            | 防范手段                                           |
| ----------------------- | ------------------------------------------------------------------- | -------------------------------------------------- |
| **DDoS 分布式拒绝服务** | 控制大量肉鸡发送海量请求，耗尽服务器带宽 / CPU / 内存，使其无法服务 | 高防 IP、CDN 流量清洗、黑洞路由、限流熔断          |
| **SYN 洪水攻击**        | 发送大量 TCP 半连接请求，不回复 ACK，占满连接队列，拒绝正常连接     | SYN Cookie、缩短超时时间、调大半连接队列           |
| **ARP 欺骗**            | 伪造 ARP 响应，篡改局域网内 IP-MAC 映射，劫持 / 监听流量            | 静态 ARP 绑定、ARP 防火墙、VLAN 隔离               |
| **DNS 劫持**            | 篡改 DNS 解析结果，将域名指向恶意 IP，跳转钓鱼网站                  | DNSSEC、使用可信公共 DNS、域名锁定、HTTPS          |
| **中间人攻击 (MITM)**   | 拦截通信双方数据，窃听、篡改、伪造信息                              | TLS/HTTPS、证书校验、VPN、数字签名                 |
| **SQL 注入**            | 拼接恶意 SQL 语句到请求中，非法查询 / 篡改 / 删除数据库             | 参数化查询、预编译语句、最小权限、WAF              |
| **XSS 跨站脚本**        | 注入恶意 JS 代码到页面，窃取 Cookie、会话、用户信息                 | 输入过滤、输出转义、CSP、Cookie HttpOnly           |
| **CSRF 跨站请求伪造**   | 利用用户已登录身份，伪造请求执行未授权操作                          | CSRF Token、Referer 校验、SameSite Cookie          |
| **文件上传漏洞**        | 上传恶意脚本 / 木马，获取服务器控制权                               | 校验文件类型 / 后缀、重命名文件、存储到非 Web 目录 |
| **弱口令攻击**          | 暴力破解 / 字典攻击简单密码，窃取账号权限                           | 强密码策略、多因素认证、登录限流、定期改密         |
